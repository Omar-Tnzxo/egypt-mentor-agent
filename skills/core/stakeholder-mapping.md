# Skill: Stakeholder Mapping

Use this skill whenever the case belongs to a domain that contains multiple actor types, buyers, users, operators, or decision-makers.

## Goal

Prevent the agent from treating a whole sector as if it had one default user or one default buyer.

## Core rule

A domain is not a person.
Before giving sector-specific guidance, identify who the current speaker is and what role they occupy inside the system.

Start from the speaker, decide from the system.
That means the agent should begin by understanding the current speaker's reality, then widen the lens to the full set of relevant actors before making an important recommendation.

## What to map

Always try to distinguish between:

- the current speaker
- the direct user
- the economic buyer
- the decision-maker
- the operator
- the beneficiary
- any blocked or ignored stakeholder whose behavior affects success

## Why this matters

The same sector can contain very different realities.
For example, in real estate the speaker may be:

- a sales agent
- a buyer
- a brokerage owner
- a sales manager
- a developer team member
- a marketer
- an operations person
- a technical builder in proptech

Each one has a different goal, pain, leverage, and success metric.

## Rules

- Do not assume one default actor for a whole domain.
- If the actor identity changes the advice materially, ask and confirm it.
- If multiple actors are involved, identify the primary actor for the current case.
- When relevant, also map secondary actors who can block or enable progress.
- Do not make a strategic judgment from the speaker's point of view alone when the system dynamics matter.

## Output requirements

When this skill is active, include:

- primary actor
- relevant secondary actors
- who gets value
- who pays
- who decides
- which actor's pain matters most right now
- whether the speaker's perspective differs from the system-level reality
